# counter-sveltekit — full-stack axum + SvelteKit Counter

This is the Phase 5 exit example: a real, full-stack application that wires
a Rust/axum server to a SvelteKit single-page app through `taut-rpc`. It
exercises every transport flavour the runtime supports — query, mutation,
and subscription — against a single piece of shared state with a real UI on
top, and stands as the reference for "how does an end-to-end taut-rpc app
actually look?"

The earlier examples (`phase1`, `phase2-auth`, `phase3-counter`,
`phase4-validate`) are surgical: each one isolates a single dimension of
the spec. This one is the integration test for the whole stack, with the
codegen, validator bridge, and SSE transport all working together against
a UI a person can actually click on.

## What it covers

- **Server** (axum, port 7712, four procedures):
  - `current() -> u64` — read-only query, returns the current value.
  - `increment(IncrementInput) -> u64` — mutation, validated `by ∈ [1, 1000]`.
  - `reset() -> u64` — mutation, no input.
  - `live() -> impl Stream<Item = u64>` — subscription, emits the current
    value on subscribe and on every subsequent change.
- **Shared state**: `Mutex<u64>` plus a `tokio::sync::broadcast::Sender<u64>`
  in a `OnceLock<AppState>`. Every mutation publishes the new value to the
  channel, and `live()` subscribes-then-snapshots for race-free fan-out.
- **Client** (SvelteKit + adapter-static, SPA-mode):
  - Imports the generated `api.gen.ts` through a `$lib/api` indirection so
    application code never touches the generated module directly.
  - Subscribes to `live()` on mount, tears down on destroy, and updates the
    UI from the broadcast stream — the +N and reset buttons mutate, then
    let the value redraw via SSE.
  - Open the page in two browser tabs; both stay in sync.

## Run sequence

The same four-step pipeline as Phase 4, plus `npm install` for the bigger
SvelteKit dependency tree. The npm runtime needs to be built once before
the client can resolve `taut-rpc` from its `file:` dependency.

### 1. Build the npm runtime (once per checkout)

```sh
cd npm/taut-rpc
npm install
npm run build
```

### 2. Build the server in IR-dump mode and run codegen

From the repository root:

```sh
cd examples/counter-sveltekit/server && cargo build && cd ../../..
cargo run -p taut-rpc-cli -- taut gen \
  --from-binary examples/counter-sveltekit/server/target/debug/counter-sveltekit-server \
  --out         examples/counter-sveltekit/client/src/lib/api.gen.ts
```

`--from-binary` re-spawns the server binary with `TAUT_DUMP_IR` set,
captures its IR, and runs the codegen step — see
`dump_if_requested(&router)` in `server/src/main.rs`. No port is bound
during the dump.

The generated `api.gen.ts` lands in `client/src/lib/` (gitignored), where
`client/src/lib/api.ts` re-exports it as a typed client.

### 3. Start the server

```sh
# terminal A
cd examples/counter-sveltekit/server
cargo run
```

The server binds `0.0.0.0:7712` and prints:

```
counter-sveltekit-server listening on http://127.0.0.1:7712
```

CORS is permissive — this is a local-only example, not a deployment
template.

### 4. Run the SvelteKit dev server

```sh
# terminal B, with the server running and api.gen.ts generated
cd examples/counter-sveltekit/client
npm install
npm run dev
```

Vite prints a localhost URL (typically `http://127.0.0.1:5173`). Open it
in a browser, then open the same URL in a second tab — clicking +1 in one
tab updates the value in both, because the server broadcasts every
mutation through the `live()` subscription.

The Vite dev server proxies `/rpc/*` to `http://127.0.0.1:7712`, so the
client's runtime URL stays origin-relative (`"/rpc"`). That keeps the
build agnostic to where the Rust server lives — switch the proxy target
and nothing else changes.

### 5. Optional: static build

```sh
cd examples/counter-sveltekit/client
npm run build
npm run preview   # serves the static export, still proxied to :7712
```

`adapter-static` materialises a flat folder of HTML/JS in `build/`. Any
HTTP server can serve it; in production you'd put SvelteKit and Rust
behind a single reverse proxy that forwards `/rpc/*` to the Rust port.

## Files

- `server/Cargo.toml`, `server/src/main.rs` — the four procedures plus
  the `OnceLock<AppState>` shared-state pattern.
- `client/package.json`, `client/svelte.config.js`,
  `client/vite.config.ts`, `client/tsconfig.json` — minimal SvelteKit
  setup with `adapter-static` (SPA-mode).
- `client/src/routes/+page.svelte` — the Counter UI.
- `client/src/routes/+layout.ts` — `ssr: false` / `prerender: false`
  toggles.
- `client/src/lib/api.ts` — thin re-export of the generated client.
- `client/src/lib/api.gen.ts` — generated by `cargo taut gen`,
  gitignored.

## What this is not

- Not a deployment template. The server has permissive CORS, no auth,
  binds `0.0.0.0`, and uses a process-global `OnceLock` for state.
- Not a `decrement` demo: the server's `Validate` impl rejects
  `by < 1`, so the UI only has a `+N` button. Add a `decrement`
  procedure if you want one.
